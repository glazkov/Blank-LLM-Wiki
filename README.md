# Шаблон проекта LLM Wiki

Этот шаблон помогает быстро запускать проекты, где агент накапливает знания в `wiki/`, а не держит их в контексте чата.

## Быстрый старт в Codex

Если у тебя пустая папка проекта в Codex, можно развернуть шаблон одним текстовым запросом.

### Запрос для разворачивания шаблона

```text
Этот каталог пустой. Разверни в нем последнюю версию шаблона из https://github.com/glazkov/Blank-LLM-Wiki, скопируй файлы шаблона в текущий проект, не переноси исходную .git, затем инициализируй новый git-репозиторий, запусти make test и make lint и кратко сообщи результат. После этого предложи следующий запрос для первичной кастомизации проекта.
```

### Следующий запрос для первичной кастомизации

```text
Прочитай README.md, AGENTS.md и страницы в wiki/operations/. Помоги провести первичную кастомизацию проекта: задай нужные вопросы, зафиксируй вводные в wiki и обнови текущее состояние проекта.
```

## Слои

- `raw/`: неизменяемые исходники.
- `wiki/`: поддерживаемая база знаний.
- `AGENTS.md`: правила работы агента.

## Первая кастомизация после копирования шаблона

1. Попросить агента собрать новые вводные по проекту.
2. Заполнить `wiki/operations/project-intake.md`.
3. Обновить `wiki/overview.md`, `wiki/operations/project-status.md` и `wiki/operations/next-steps.md`.
4. Только после этого переходить к обработке первых источников и обычной работе.

Обычно нужно уточнить:

- предмет проекта и его границы;
- какие типы источников будут поступать в `raw/`;
- какие артефакты должны появляться в `wiki/` и `outputs/`;
- какие есть ограничения, критерии качества и открытые вопросы.

## Базовый поток работы

1. Сначала провести первичную кастомизацию проекта.
2. Добавить новый источник в `raw/inbox/` или `raw/sources/`.
3. Попросить агента обработать источник и обновить `wiki/`.
4. Запускать `make lint` после заметных изменений.
5. Обновлять `wiki/operations/project-intake.md`, `wiki/operations/project-status.md` и `wiki/operations/next-steps.md`, чтобы проект можно было восстановить после остановки.
6. Фиксировать значимые изменения в git.

## Команды

- `make lint`: полная проверка wiki.
- `make lint-struct`: структурная проверка.
- `make lint-semantic`: семантическая проверка.
- `make status`: краткий статус проекта и текущая точка продолжения.
- `make test`: запуск локальных проверок шаблона.

## Восстановление состояния

- `wiki/operations/project-intake.md`: исходные вводные, границы проекта и правила его кастомизации.
- `wiki/operations/project-status.md`: текущее состояние проекта, последний завершенный этап и активный фокус.
- `wiki/operations/next-steps.md`: ближайшие шаги, блокеры и варианты продолжения.
- Если ближайший шаг пока не определен, в `wiki/operations/next-steps.md` должен быть явный вопрос пользователю или 2-3 конкретных варианта продолжения.
- Если следующий шаг неочевиден, агент должен либо спросить пользователя, либо предложить варианты и записать их в `wiki/operations/next-steps.md`.

## Правило ответа

Для нетривиальных запросов агент сначала перечисляет допущения и что может сильно изменить ответ.

## Языковое правило

Долговечные тексты в этом проекте пишутся по-русски. Английские слова допустимы только для точных технических имен, команд, путей и терминов, которые нельзя корректно перевести.

## История изменений

После копирования шаблона новый проект нужно сразу инициализировать через `git init`, если история еще не создана. Значимые изменения дальше тоже нужно фиксировать в git, чтобы история оставалась журналом эволюции знаний.
